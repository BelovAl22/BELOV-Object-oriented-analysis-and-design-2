В рамках лабораторной работы было разработано приложение-симулятор автомобиля, демонстрирующее применение структурного паттерна **«Хранитель» (Memento)**. Архитектура системы построена в соответствии с классической реализацией данного паттерна и включает четыре ключевых класса: `Car`, `CarState`, `HistoryManager` и `App`.

Класс `Car` выполняет роль **Originator (Создателя)**. Он хранит текущее состояние объекта, включающее такие параметры, как мощность двигателя, масса автомобиля, тип привода, коэффициент аэродинамического сопротивления и время переключения передач. Помимо хранения состояния, класс реализует бизнес-логику симуляции, позволяя вычислять время разгона от 0 до 100 км/ч и максимальную скорость. Также он предоставляет методы `save()` и `restore()`, предназначенные для создания снимка состояния и его последующего восстановления.

Класс `CarState` представляет собой **Memento (Хранитель состояния)**. Он используется для сохранения состояния объекта `Car` в конкретный момент времени. Данный класс содержит только данные (поля состояния) и не включает в себя какой-либо бизнес-логики. Это позволяет обеспечить простоту и безопасность хранения состояния, а также исключает возможность его случайного изменения извне.

Класс `HistoryManager` выполняет роль **Caretaker (Опекуна)**. Он отвечает за управление историей состояний объекта. Внутри класса реализованы два стека: `undo_stack` и `redo_stack`, которые используются для поддержки операций отмены и повторения действий. Важно отметить, что `HistoryManager` не имеет доступа к внутренней структуре объекта `Car` и работает исключительно с объектами `CarState`. Это позволяет сохранить инкапсуляцию и уменьшить связность компонентов системы.

Класс `App` представляет собой клиентскую часть приложения и реализует графический пользовательский интерфейс. Он обеспечивает взаимодействие пользователя с системой: изменение параметров автомобиля, запуск симуляции и выполнение операций Undo/Redo. В обновлённой версии приложения сохранение состояния выполняется автоматически при запуске симуляции, что устраняет необходимость ручного сохранения. Для предотвращения дублирования состояний используется дополнительное поле, в котором хранится последнее сохранённое состояние и выполняется его сравнение с текущим.

# Взаимодействие классов

Взаимодействие компонентов системы организовано следующим образом:

- Класс `Car` создаёт объект `CarState` при вызове метода `save()`, тем самым формируя снимок своего состояния.
- Класс `HistoryManager` сохраняет полученные объекты `CarState` и управляет их последовательностью, реализуя историю изменений.
- При запуске симуляции в классе `App` происходит автоматическое сохранение состояния, если оно отличается от предыдущего.
- При выполнении операций Undo/Redo объект `CarState` передаётся обратно в `Car`, который восстанавливает своё состояние с помощью метода `restore()`.
- Класс `App` выступает посредником между пользователем и системой, инициируя вызовы методов и обновляя пользовательский интерфейс.

# Преимущества использования паттерна Memento

Применение паттерна «Хранитель» в данной работе обеспечивает следующие преимущества:

- **Сохранение инкапсуляции**: внешний код не имеет доступа к внутренней структуре класса `Car`, а работает только с объектами состояния.
- **Слабая связность компонентов**: изменения в структуре класса `Car` не требуют модификации класса `HistoryManager`.
- **Гибкость и расширяемость**: система легко расширяется за счёт добавления новых параметров состояния без изменения логики управления историей.
- **Автоматизация работы с историей**: пользователь освобождается от необходимости вручную сохранять состояния, что повышает удобство использования.
- **Надёжная реализация Undo/Redo**: механизм отката реализован корректно и не зависит от конкретной реализации объекта.

Применение данного паттерна продемонстрировало его эффективность при разработке систем, требующих сохранения и восстановления состояния объектов.
