# Лабораторная работа: паттерн Адаптер (Adapter)

## 1. Описание проблемы

Необходимо разработать программную систему для сравнения характеристик автомобилей из разных регионов. Приложение должно позволять пользователю выбирать два автомобиля (японские или американские), сравнивать их по ключевым параметрам (скорость, объём двигателя, расход топлива, клиренс, разгон, цена, вес) и определять победителя по сумме очков.
Основная сложность заключается в том, что японские и американские автомобили используют разные системы измерений и различные имена методов для получения характеристик:
Японские автомобили используют метрическую систему (км/ч, литры, л/100км, мм, кг)
Американские автомобили используют имперскую систему (mph, куб.дюймы, MPG, дюймы, фунты)
Без применения паттерна разработчики столкнулись бы с необходимостью писать отдельную логику сравнения для каждого типа автомобилей. Клиентский код вынужден был бы проверять тип объекта и выполнять конвертацию единиц вручную, что приводит к дублированию кода и нарушению принципа открытости/закрытости. При добавлении нового региона (например, европейские автомобили) потребовалось бы переписывать функцию сравнения.
## 2. Решение: Структура программы

Система реализована на языке C++ с использованием объектно-ориентированного подхода и паттерна Адаптер. Архитектура приложения строится на взаимодействии нескольких ключевых классов, описанных в соответствии со стандартом UML.
**Основные классы системы:**
* ICarAdapter (Целевой интерфейс) – абстрактный класс, определяющий единый контракт для всех автомобилей. Содержит виртуальные методы для получения характеристик в метрической системе (getMaxSpeedKmh, getEngineVolume, getFuelConsumption и др.).
* JapaneseCar (Японский автомобиль) – адаптируемый класс. Представляет японский автомобиль с данными в метрической системе. Содержит методы getSpeedKmh, getEngineLiters, getFuelLPer100 и другие.
* JapaneseAdapter (Японский адаптер) – адаптер для японских автомобилей. Оборачивает объект JapaneseCar и делегирует вызовы методов, обеспечивая соответствие интерфейсу ICarAdapter.
* AmericanCar (Американский автомобиль) – адаптируемый класс. Представляет американский автомобиль с данными в имперской системе. Содержит методы getSpeedMph, getEngineCubicInches, getFuelMPG и другие.
* AmericanAdapter (Американский адаптер) – адаптер для американских автомобилей. Оборачивает объект AmericanCar, делегирует вызовы и выполняет конвертацию единиц измерения (mph в км/ч, куб.дюймы в литры, MPG в л/100км и др.).
* CarFactory (Фабрика) – класс, инкапсулирующий логику создания объектов. Возвращает указатель на ICarAdapter, скрывая от клиента конкретные классы адаптеров.
* CarComparator (Компаратор) – класс, реализующий алгоритм сравнения двух автомобилей. Работает только с интерфейсом ICarAdapter, не зная о конкретных реализациях.
* WndProc (Обработчик окон) – главный класс, управляющий графическим интерфейсом WinAPI, обработкой сообщений и жизненным циклом приложения.

Взаимодействие объектов:
Клиентская часть (WndProc) работает с объектами через общий интерфейс ICarAdapter. При выборе автомобиля фабрика создаёт соответствующий адаптер, который оборачивает адаптируемый объект. Адаптер американского автомобиля выполняет конвертацию единиц измерения при каждом вызове метода. Компаратор получает два объекта ICarAdapter и сравнивает их характеристики, не зная о том, какие классы находятся внутри. Удаление объектов осуществляется в деструкторе адаптера, который освобождает память адаптируемого объекта.

## 4. Сравнение подходов

### 4.1 Без использования паттерна

В альтернативной реализации клиентский код вынужден работать с конкретными классами JapaneseCar и AmericanCar напрямую. Для сравнения автомобилей требуется проверка типа объекта с помощью dynamic_cast или флагов. Конвертация единиц измерения выполняется в функции сравнения, что приводит к дублированию кода при каждом вызове.
При добавлении нового типа автомобиля (например, EuropeanCar) необходимо модифицировать функцию сравнения, добавляя новые ветви условных операторов. Это нарушает принцип открытости/закрытости и увеличивает риск ошибок при изменении существующего кода.

### 4.2 С использованием паттерна Адаптер

Текущая реализация позволяет клиенту работать с единым интерфейсом ICarAdapter. Конвертация единиц измерения инкапсулирована внутри адаптеров и не видна клиентскому коду. Функция сравнения остаётся неизменной при добавлении новых типов автомобилей.
Для интеграции нового региона достаточно создать новый адаптируемый класс и соответствующий адаптер, реализующий ICarAdapter. Код фабрики, компаратора и интерфейса не требует изменений.

## 5. Вывод

В ходе работы была разработана система сравнения автомобилей, соответствующая требованиям лабораторной работы. Применение паттерна Адаптер позволило:
1. Унифицировать интерфейс – все автомобили обрабатываются через единый интерфейс ICarAdapter, независимо от региона происхождения.
2, Инкапсулировать конвертацию – преобразование единиц измерения вынесено в адаптеры, клиентский код работает только с метрической системой.
3. Обеспечить расширяемость – добавление нового региона требует создания только двух классов (адаптируемый класс и адаптер), без изменения существующего кода сравнения.
4. Соблюсти принцип инверсии зависимостей – высокоуровневые модули (CarComparator) зависят от абстракции (ICarAdapter), а не от деталей реализации.
5. Гарантировать безопасность памяти – адаптер владеет указателем на адаптируемый объект и освобождает память в деструкторе.

Система успешно проходит тестирование: корректно сравнивает автомобили разных регионов, определяет победителя по сумме очков и выводит результат в графический интерфейс. Применение паттерна Адаптер сделало архитектуру гибкой, масштабируемой и удобной для поддержки.
